# CloudBase 规范工作流 Power

## 概述

这个 Power 提供了标准的软件工程工作流框架，用于需求分析、技术设计和任务规划，确保 CloudBase 项目的高质量开发过程。

## 适用场景

### 何时使用此 Power

**适用于结构化开发工作流**，当你需要：
- 从零开始开发新功能
- 设计复杂的系统架构
- 集成多个模块和服务
- 涉及数据库设计的项目
- 涉及 UI 设计的项目
- 确保高质量的需求分析和验收标准

### 何时不使用此 Power

**不适用于以下情况：**
- 简单的错误修复
- 文档更新
- 配置更改
- 代码重构（除非明确要求）

## 工作流程概述

### 四阶段开发流程

1. **需求文档和验收标准设计**
2. **技术方案设计**
3. **任务分解和规划**
4. **任务执行和跟踪**

**重要：每个阶段都必须经过用户确认才能进入下一阶段**

## 主要功能

### 📋 需求分析
- 使用 EARS（Easy Approach to Requirements Syntax）方法
- 标准化需求描述语法
- 清晰的验收标准定义

### 🏗️ 技术设计
- 全面的解决方案架构规划
- 技术栈选择和评估
- 数据库和接口设计

### 📊 任务管理
- 详细的任务分解
- 进度跟踪和状态更新
- 独立高效的任务执行

### 🎨 集成设计
- 与 UI 设计规则无缝集成
- 数据库设计工作流整合
- 多模块协调开发

### ✅ 质量保证
- 内置验收标准和验证流程
- 一致性和完整性检查
- 高质量交付保障

## 详细工作流程

### 阶段 1：需求文档和验收标准设计

使用 EARS 方法完成需求设计。如果需求涉及前端页面，**必须严格参考 UI 设计规则**。

**EARS 语法格式：**
```
While <可选前置条件>, when <可选触发条件>, the <系统名称> shall <系统响应>
```

**示例：**
- When "静音" 被选择时，笔记本电脑应抑制所有音频输出

保存到 `specs/spec_name/requirements.md`，用户确认后进入下一阶段。

### 阶段 2：技术方案设计

基于当前技术架构和确认的需求，设计技术方案。应简洁但准确描述：
- 技术架构
- 技术栈选择
- 数据库/接口设计
- 测试策略
- 安全考虑

必要时使用 Mermaid 图表。保存到 `specs/spec_name/design.md`。

### 阶段 3：任务分解

基于需求文档和技术方案，分解具体任务。

**任务格式：**
```markdown
# 实施计划

- [ ] 1. 任务信息
  - 具体要做的事情
  - ...
  - _需求：相关需求点编号
```

保存到 `specs/spec_name/tasks.md`。

### 阶段 4：任务执行

- 开始正式任务执行
- 实时更新 `tasks.md` 文件中的任务状态
- 独立自主地工作
- 确保效率和质量
- 完成时标记任务为已完成

## 核心原则

### 🔒 用户确认必需
每个阶段都必须经过用户确认才能继续

### 📝 EARS 语法
使用 EARS 方法进行需求描述

### 🎨 UI 设计集成
当需求涉及前端页面时，必须参考 UI 设计规则并在需求阶段确定设计风格

### 🔧 技术准确性
技术方案应简洁但准确

### 📊 任务跟踪
在整个执行过程中保持任务状态更新

### 🚀 独立执行
在保持质量的同时自主工作

## 使用方法

1. **激活 Power**: 使用 Kiro Powers 激活此 Power
2. **启动工作流**: 当收到新需求时，按照四阶段流程执行
3. **用户确认**: 每个阶段完成后等待用户确认
4. **持续跟踪**: 在执行阶段持续更新任务状态
5. **质量保证**: 确保每个阶段的输出质量

- 保持与用户的及时沟通
- 维护清晰的任务状态
- 注重质量而非速度
- 适当使用可视化工具（Mermaid 图表）

通过这个 Power，你可以确保项目开发过程的标准化和高质量交付。
